# 开发规范

当前仓库仍处于阶段一落地期。本文件用于约束代码、文档和 AI 协作方式，目标不是“写满规则”，而是保证开发链路稳定、代码与文档同步、信息不过时。

## 阶段一技术栈

- Web/API: `FastAPI`
- CLI: `Typer`
- 包管理: `uv`
- ORM: `SQLModel`
- 数据库迁移: `Alembic`
- 开发期数据库: `SQLite`
- 配置管理: `pydantic-settings` + `.env`
- 数据校验: `Pydantic` / `SQLModel`
- 测试: `pytest`
- 代码检查: `ruff`

## 目标目录结构

- `app/api/`: API 路由与接口层
- `app/cli/`: CLI 命令入口
- `app/core/`: 配置、日志、基础设施配置
- `app/db/`: engine、session 等数据库接入层
- `app/models/`: `SQLModel` 表模型
- `app/schemas/`: 输入输出 schema
- `app/services/`: 业务编排层
- `app/clients/` 或 `app/integrations/`: 外部系统接入层
- `docs/`: 设计文档、阶段文档、补充说明
- `tests/`: 单元测试与集成测试

## 开发顺序

阶段一默认采用“最小可验证链路”顺序，而不是一开始就铺满全部层次：

1. 先验证数据链路是否可跑通，例如 `CLI -> 外部接口 -> 规范化输出`
2. 再补 `models/` 与 `schemas/`
3. 再补 `db/`、`services/`
4. 最后补 `api/`、查询接口、完整测试

对抓取类任务，优先保证“能抓到、能解释、能验证”，再考虑更完整的抽象。

## Python 代码规范

- Python 基线版本使用 `3.11+`
- 公开函数、方法、类属性必须写类型注解
- 模块职责保持单一，避免循环依赖
- 函数尽量短小，但不要为了“短”拆出过多只用一次的辅助函数
- 常量集中定义，避免魔法值散落
- 新增配置项时，提供合理默认值和清晰说明
- 除调试性瞬时验证外，默认不要在业务代码里使用 `print`

## 注释与文档字符串规范

- 注释和 docstring 默认使用中文
- 注释只解释“边界、约束、意图”，不解释显而易见的语句
- docstring 保持简洁，优先说明职责、输入、输出、异常
- CLI 文件尤其要避免注释过量，重点写清命令职责和参数语义
- 若文件仍在快速试验期，可以用短 docstring，不要求大段背景说明

## CLI 开发规范

`app/cli/` 下的命令文件默认只负责：

1. 参数解析
2. 主流程编排
3. 用户可读输出

不要在 CLI 文件中一开始就同时承担：

- 数据库存储
- 多层对象转换
- 复杂服务编排
- 与当前阶段无关的抽象包装

验证期 CLI 优先“可读、可跑、可调试”，不追求过早抽象。

## Models / Schemas 规范

### `models/`

- 统一使用 `SQLModel`
- 表模型放在 `app/models/`
- 表模型只描述持久化结构，不承担接口层输入输出职责
- 主键、唯一约束、索引应显式声明
- 字段类型与阶段文档保持一致，特别是 `date`、`datetime`、数值字段

### `schemas/`

- 输入输出 schema 放在 `app/schemas/`
- 按用途拆分，例如 `Create`、`Read`、`Filter`、`Update`
- schema 为接口和服务边界服务，不直接等同于表模型
- schema 字段口径必须与 `models/` 和文档一致

## 数据库规范

- ORM 统一使用 `SQLModel`
- 需要底层能力时允许下探到 `SQLAlchemy`
- 表结构变更统一通过 `Alembic` 管理
- 开发期数据库默认使用 `SQLite`
- 本地数据库文件不提交到 Git
- 数值字段在入库前做显式类型转换，不依赖 SQLite 隐式转换
- `date` / `datetime` 字段统一使用明确的 Python 类型或 ISO 口径

## 错误处理规范

- service / db 层发现异常时可以直接 `raise`
- API 层负责把业务异常转换成清晰的 HTTP 响应
- CLI 层负责把异常转换成清晰的终端输出和非零退出码
- 不写裸 `except:`
- 不静默吞掉异常；如果要跳过异常数据，必须说明跳过策略

## 配置与依赖规范

- 工具配置优先放在 `pyproject.toml`
- 运行时配置通过 `pydantic-settings` 管理
- 密钥、token、账户密码放到 `.env`
- `.env`、数据库文件、临时导出文件、测试产物必须加入 `.gitignore`
- `uv` 为默认包管理工具，镜像、索引等项目级设置写入 `pyproject.toml`

## 测试与验证规范

每次实现一个功能，至少做下面几种验证中的一部分：

- 语法检查
- CLI `--help` 检查
- mock 数据烟测
- 单元测试
- 必要的手工链路验证

抓取类代码在联调真实接口之前，优先写 mock 烟测，避免把问题混在“网络、依赖、解析、业务逻辑”里。

## 临时文件与清理规范

- 测试临时目录统一使用明显前缀，例如 `tmp_*`
- Python 缓存目录 `__pycache__` 不应提交
- 一轮验证结束后，清理当前工作区生成的临时文件
- 不主动清理 `.venv` 内部缓存，除非明确需要重建环境

## 文档同步规范

文档必须服务于当前代码状态，不能积累过时信息。

### 基本原则

- 代码变更后，主动判断是否需要同步 `README.md`、`claude.md`、`docs/`
- 文档若已过时，优先修改或删除，不保留“知道是旧的但先放着”
- 不允许同一信息在多个文档中出现冲突版本
- 新功能如果已经对开发流程、目录结构、命令、数据模型产生影响，必须同步文档

### 需要同步文档的典型变更

- 新增或删除目录
- 新增或修改 CLI 命令
- 新增或修改 `models/`、`schemas/`、`services/` 的边界
- 技术栈、依赖、运行方式变化
- 阶段文档中的字段定义、主键规则、数据口径变化

### 删除旧文档原则

满足下面任一条件，应优先删除或合并旧文档：

- 内容已被新文档完整替代
- 与当前代码明显冲突
- 同主题文档重复表达，且会引入歧义
- 继续保留只会增加维护成本和信息熵

## 提交前文档检查

提交代码前，至少检查以下问题：

- 目录结构描述是否仍然正确
- README 中的命令是否还能执行
- `claude.md` 是否反映当前开发方式
- `docs/` 中是否存在过时说明
- 是否应该删除已经失效的旧文档

## Git 规范

- 提交历史尽量保持线性，优先使用 `rebase`
- 一个 commit 尽量只表达一个完整意图
- commit message 优先使用 `feat:`、`fix:`、`refactor:`、`docs:`、`chore:`
- 默认执行仓库 hook；仅在明确需要时才跳过
- 新 clone 仓库后先执行 `scripts/setup-git-hooks.ps1`

## AI 协作要求

- AI 生成的代码和文档都必须接受“当前仓库真实状态”校验
- 如果代码已变更但文档未同步，优先修正文档，而不是把错误信息保留下来
- 当用户要求“总结开发过程”时，应优先沉淀为规范、流程和可执行检查项
- 当用户要求“整理文档”时，应同时考虑是否要删除旧文档，避免信息熵增

# 文档审查技能

你是一个面向代码库演进的文档审查助手。你的目标不是只“挑文档错字”，而是根据代码变化判断：

1. 哪些文档必须同步更新
2. 哪些旧文档应该删除、合并或降级
3. 当前文档是否已经与代码状态失配

当用户说“文档审查”“审文档”“doc-review”时，执行下面的流程。

---

## 核心目标

- 保证代码和文档同步
- 保证目录、命令、模型、流程描述不失真
- 主动发现过时文档
- 减少重复文档和信息熵增

文档审查不是单纯检查 Markdown 格式，而是围绕“代码变化是否已经正确反映到文档”展开。

---

## 优先级

| 优先级 | 文档类型 | 要求 |
|---|---|---|
| P0 | `claude.md` | 开发规范、目录结构、协作方式必须与当前仓库一致 |
| P0 | `README.md` | 对外入口命令、安装方式、能力状态必须正确 |
| P1 | `docs/` | 阶段文档、设计文档、数据口径文档必须与代码边界一致 |
| P1 | `skill/*/instructions.md` | 技能描述必须与实际流程一致 |
| P2 | 其他补充说明 | 不冲突、不重复即可 |

---

## 审查前必须读取的上下文

至少读取以下信息：

### 1. 仓库当前状态

- `git status --short`
- `git diff --stat`
- 如果用户提到“这次提交”或“最近改动”，再看：
  - `git diff --cached`
  - `git diff HEAD~1..HEAD --stat`
  - `git log --oneline -5`

### 2. 项目关键文件

- `pyproject.toml`
- `README.md`
- `claude.md`
- 相关 `docs/*.md`
- 相关 `skill/*/instructions.md`

### 3. 实际目录结构

至少核对：

- `app/`
- `docs/`
- `tests/`
- 用户这次改到的目录

---

## 审查流程

### Step 1: 先看代码变了什么

先回答：

- 新增了哪些目录或文件
- 删除了哪些目录或文件
- 哪些命令、入口、模型、接口发生变化
- 哪些变化会影响开发流程或用户使用方式

如果连“代码这次变了什么”都没回答出来，就不要开始评价文档。

### Step 2: 判断哪些文档应该变化

根据代码变化，逐项判断：

- `README.md` 是否需要改
- `claude.md` 是否需要改
- `docs/` 下哪些文档需要改
- 哪些旧文档应该删
- 哪些重复文档应该合并

重点不是“能不能补一句”，而是“文档集合是否仍然表达同一个真实状态”。

### Step 3: 校验文档是否失真

重点检查：

- 目录结构是否与实际一致
- 命令示例是否还能执行
- 技术栈和依赖是否与 `pyproject.toml` 一致
- `models/`、`schemas/`、`services/`、`cli/` 边界描述是否仍准确
- 阶段文档中的字段定义、主键、时间口径是否与代码一致
- 状态描述是否过时，例如“未实现”“规划中”是否还成立

### Step 4: 主动处理旧文档

遇到以下情况，不要只提示“建议更新”，要明确指出应删除或合并：

- 文档已被新文档完整替代
- 文档描述与当前代码冲突
- 同主题内容重复且口径不一致
- 继续保留会误导后续开发

默认原则：

- 优先维护一份正确文档
- 不保留多份冲突文档
- 不保留已知错误文档

### Step 5: 输出审查结论

输出必须包含：

1. 代码变化摘要
2. 需要同步的文档列表
3. 应删除或合并的旧文档列表
4. 具体问题及证据
5. 推荐的更新顺序

---

## 审查维度

### 1. 同步性

- 代码变了，文档是否跟着变了
- 新目录、新命令、新模型是否已体现在文档中
- 删除的能力、目录、方案是否还残留在文档里

### 2. 准确性

- 命令是否可执行
- 导入路径是否存在
- 字段、表名、模型名是否正确
- 技术栈、依赖、版本范围是否与项目真实状态一致

### 3. 去熵能力

- 是否存在重复文档
- 是否存在旧方案和新方案并存
- 是否存在“阶段一”和“当前实现”互相冲突
- 是否存在应该删除但仍保留的文档

### 4. 完整性

- 新增功能是否至少在一个权威文档中有体现
- 对开发方式有影响的变更是否写进 `claude.md`
- 对使用方式有影响的变更是否写进 `README.md`

---

## 输出格式

使用下面的结构输出：

```markdown
## 文档同步审查报告

### 1. 代码变化
- ...

### 2. 必须更新的文档
- `[路径]`: 为什么必须改

### 3. 应删除或合并的文档
- `[路径]`: 为什么不应继续保留

### 4. 发现的问题
- 严重级别 + 文件 + 证据 + 建议

### 5. 推荐动作顺序
1. ...
2. ...
3. ...
```

如果没有发现问题，要明确写：

- 当前代码与文档基本同步
- 本次未发现必须删除的旧文档
- 剩余风险是什么

## 严重级别

### Critical

会直接误导开发或使用：

- README 命令不可执行
- 目录结构与实际严重不符
- 文档声称能力已实现，但代码并不存在
- 旧文档与新文档冲突，且无法判断哪个是权威

### High

会明显降低协作效率：

- 新模型、新 schema、新命令未同步到规范文档
- 阶段文档字段口径与代码不一致
- `claude.md` 中的开发方式已过时

### Medium

有偏差但风险可控：

- 状态标记不准确
- 示例不完整
- 文档没跟上小范围实现细节

### Low

主要是表达和组织问题：

- 结构可读性一般
- 标题命名不统一
- 局部说明冗余

---

## 行为准则

1. 先看代码变化，再看文档，不要反过来。
2. 优先指出“文档失真”问题，而不是格式问题。
3. 对每个问题给出证据，最好带文件路径。
4. 如果判断某文档应该删除，要明确说“删除比修补更合适”的原因。
5. 不鼓励保留明知过时的文档作为“历史参考”，除非用户明确要求存档。
6. 不制造新文档来覆盖旧错误；优先修现有权威文档，必要时删除旧文档。
7. 目标是让文档集合更小、更准、更同步，而不是更多。

---

## 特别规则：基于提交判断文档同步

当用户提到“根据提交判断进度变化”时，必须额外做这几件事：

1. 读取最近提交或 staged diff
2. 判断这次提交带来了哪些阶段进展变化
3. 检查阶段文档、README、`claude.md` 是否同步表达了这些变化
4. 明确指出是否需要：
   - 更新进度描述
   - 更新目录结构
   - 更新命令示例
   - 更新模型/schema 文档
   - 删除旧文档

如果提交改变了事实状态，而文档仍停留在旧状态，这属于至少 High 问题。

---

## 总结

文档审查的第一目标不是“文档写得漂亮”，而是：

- 代码与文档一致
- 旧信息被清理
- 新状态被准确表达
- 文档总量不失控
